Saya perlu audit Binance FAPI testnet endpoints untuk trading ML pipeline. Tolong:

1. MARKET DATA ENDPOINTS (No Auth Required)

a) GET https://demo-fapi.binance.com/fapi/v1/exchangeInfo
   Purpose: Verify symbol filters (LOT_SIZE maxQty, MIN_NOTIONAL, PRICE_FILTER tickSize, PERCENT_PRICE ±5% bounds)
   Test symbols: WIFUSDT, SOLUSDT, BTCUSDT, KMNOUSDT
   Expected: Each symbol has filters array with filterType: PRICE_FILTER, LOT_SIZE, MIN_NOTIONAL, PERCENT_PRICE, MARKET_LOT_SIZE
   Catat:
     - tickSize per symbol (e.g., WIFUSDT: integer-only vs BTCUSDT: 0.01?)
     - maxQty dari LOT_SIZE dan MARKET_LOT_SIZE (mana lebih kecil?)
     - minNotional value (should be ≥5 on testnet)
     - multiplierUp/Down dari PERCENT_PRICE (expecting ±1.05 / 0.95)

b) GET https://demo-fapi.binance.com/fapi/v1/klines?symbol=BTCUSDT&interval=15m&limit=100
   Purpose: Verify OHLCV bar structure for 15-minute candles
   Expected: Array of 100 bars, each with [open_time_ms, open, high, low, close, volume, close_time_ms, ...]
   Catat:
     - open_time format (milliseconds? e.g., 1702500900000)
     - close price realistic for BTCUSDT (40k-46k range expected 2026-04)
     - volume > 0

c) GET https://demo-fapi.binance.com/fapi/v1/premiumIndex?symbol=SOLUSDT
   Purpose: Get funding rate and mark price
   Expected: JSON with fields: symbol, markPrice, lastFundingRate, nextFundingTime
   Catat:
     - lastFundingRate (numeric, e.g., 0.0001 = 0.01% per 8h)
     - markPrice (realistic vs current SOLUSDT price?)

2. ACCOUNT & POSITION DATA (Signed Requests)

Setup: Add to Postman Authorization tab:
  Type: Custom
  Header: X-MBX-APIKEY: <your BINANCE_DEMO_API_KEY>
  Pre-request script:
    const timestamp = Date.now();
    const recvWindow = 5000;
    pm.environment.set("timestamp", timestamp);
    pm.environment.set("recvWindow", recvWindow);

a) GET https://demo-fapi.binance.com/fapi/v2/account
   (Add to URL params: ?timestamp={{timestamp}}&recvWindow={{recvWindow}}&signature=<HMAC-SHA256 of query string>)
   Purpose: Get wallet balance and account status
   Expected: JSON with fields: totalWalletBalance, availableBalance, totalInitialMargin, positions[]
   Catat:
     - totalWalletBalance (actual demo account balance, e.g., 5000 USDT)
     - availableBalance (free margin = totalWalletBalance - totalInitialMargin)
     - Field names (are they camelCase? e.g., "totalWalletBalance" not "total_wallet_balance"?)

b) GET https://demo-fapi.binance.com/fapi/v2/positionRisk?symbol=SOLUSDT
   (Add to URL params: ?symbol=SOLUSDT&timestamp={{timestamp}}&recvWindow={{recvWindow}}&signature=...)
   Purpose: Get position details (entry price, PnL, liquidation price)
   Expected: Array of position objects with fields: symbol, positionAmt, entryPrice, unRealizedProfit, markPrice, liquidationPrice
   Catat:
     - Response is ARRAY (not single object)
     - positionAmt field (code depends on this, not "amount")
     - If no position: empty array []
     - markPrice format (numeric string or float?)

c) GET https://demo-fapi.binance.com/fapi/v2/openOrders
   (Add to URL params: ?timestamp={{timestamp}}&recvWindow={{recvWindow}}&signature=...)
   Purpose: List all pending orders
   Expected: Array of order objects with fields: symbol, orderId, status (NEW/FILLED/CANCELED), type (MARKET/LIMIT), side (BUY/SELL), price, qty
   Catat:
     - Response is ARRAY (empty if no orders)
     - Status values (NEW, FILLED, CANCELED, etc.)
     - orderId format (integer, e.g., 12345678)

3. ORDER PLACEMENT (Signed POST)

a) POST https://demo-fapi.binance.com/fapi/v1/order
   Method: POST
   Headers: X-MBX-APIKEY: <API_KEY>
   Body (form-data or x-www-form-urlencoded):
     - symbol: SOLUSDT
     - side: BUY
     - type: MARKET
     - quantity: 1.0
     - timeInForce: GTC
     - timestamp: {{timestamp}}
     - recvWindow: 5000
     - signature: <HMAC-SHA256>

   Purpose: Place market BUY order (entry signal)
   Expected: JSON with orderId, status (FILLED), avgPrice, executedQty
   Catat:
     - Does order FILL immediately? (market orders on testnet should be instant)
     - avgPrice field present? (code uses this, not "price")
     - executedQty matches requested quantity?
     - No error -1111 (precision error)?

b) POST https://demo-fapi.binance.com/fapi/v1/order
   (Same as above, but for TP/SL bracket)
   Body:
     - symbol: SOLUSDT
     - side: SELL
     - type: LIMIT  ← IMPORTANT: NOT TAKE_PROFIT_MARKET (testnet doesn't support it)
     - price: 140.00
     - stopPrice: 140.00
     - quantity: 1.0
     - timeInForce: GTC
     - reduceOnly: true
     - timestamp: {{timestamp}}
     - signature: <HMAC-SHA256>

   Purpose: Place TP bracket order (take-profit)
   Expected: orderId, status=NEW (waiting to fill)
   Catat:
     - Price within ±5% PERCENT_PRICE band from mark price? (e.g., if mark=135, then price must be 128-142)
     - If error -4120 "use Algo Order API": This is EXPECTED (reason for LIMIT fallback in code)
     - Order appears in GET /fapi/v2/openOrders?

c) DELETE https://demo-fapi.binance.com/fapi/v1/order
   Method: DELETE
   Body:
     - symbol: SOLUSDT
     - orderId: <from previous POST response>
     - timestamp: {{timestamp}}
     - signature: <HMAC-SHA256>

   Purpose: Cancel stale limit order
   Expected: Cancelled order details
   Catat:
     - status: CANCELED
     - Order no longer in openOrders?

4. LEVERAGE (Signed POST)

a) POST https://demo-fapi.binance.com/fapi/v1/leverage
   Body:
     - symbol: SOLUSDT
     - leverage: 2
     - timestamp: {{timestamp}}
     - signature: <HMAC-SHA256>

   Purpose: Set symbol leverage
   Expected: JSON with leverage: 2, maxNotionalValue: <notional cap>
   Catat:
     - maxNotionalValue (max position notional allowed at 2× leverage, e.g., 500000 USDT)

b) GET https://demo-fapi.binance.com/fapi/v1/leverageBracket?symbol=BTCUSDT
   (Signed)
   Purpose: View leverage bracket tiers
   Expected: Array with brackets[] objects showing leverage tiers and notional caps
   Catat:
     - Number of brackets (expect 3-5 tiers)
     - Bracket structure (notionalFloor, notionalCap, maintMarginRatio)
